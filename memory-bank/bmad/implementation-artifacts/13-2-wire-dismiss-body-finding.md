# Story 13.2: `ai-pr-review dismiss` CLI subcommand + wire `dismiss-body-finding`

## Story

As a **maintainer of ai-pr-review**,
I want the `/ai-pr-review dismiss|false-positive|wont-fix` command (posted as a top-level PR comment) handled by a new `ai-pr-review dismiss` CLI subcommand instead of ~650 lines of untested inline bash/GraphQL,
so that the `dismiss-body-finding` job's classification, thread-resolution, and dismissal logic runs the pytest-covered Python module built in story 13-1, and the bash orchestration this epic exists to retire actually stops running for the first (and richest) of its four jobs.

This is phase 2 of a 4-phase epic (Epic 13, GitHub issue #554). Story 13-1 shipped the `ai_pr_review/slash/dismiss.py` module and promoted `GitHubProvider` primitives with **zero production call sites** — `dismiss_by_finding_id`/`dismiss_inline_reply` are fully implemented and unit-tested but unreachable from the CLI. This story adds the CLI entry point and rewires exactly one job (`dismiss-body-finding`) to use it. `dismiss-finding` (13-3) and `feedback-command`'s two dismiss-adjacent steps (13-4) are separate stories.

## Acceptance Criteria

1. A new `ai-pr-review dismiss` Click subcommand is added to `ai_pr_review/cli.py`, alongside (not replacing) the existing `slash` subcommand. It follows the `slash` command's established pattern: Click options with `envvar=` bindings reading from `SLASH_*` env vars, not new `ReviewConfig` fields (verified: `ReviewConfig.from_env()` at `config.py:399-536` reads no `SLASH_*` var today, and `_KNOWN_AI_VARS` at `config.py:23-86` has no `SLASH_*` entries — this command must not add any).
2. The new subcommand reads: `--finding-id` (`envvar=SLASH_FINDING_ID`, optional int — absence means "list active IDs" mode, matching the existing bash job's `lookup` step at `slash-commands.yml:431-669`), `--actor` (`envvar=SLASH_ACTOR`, required), `--command` (`envvar=SLASH_COMMAND`, required, one of `dismiss`/`false-positive`/`wont-fix`), `--pr-number` (`envvar=SLASH_PR_NUMBER`, required int).
3. The subcommand builds a `GitHubProvider` for the dismiss call. Because `provider_from_env()` (`vcs/__init__.py:68-79`) returns the general `VcsProvider` union type and `dismiss_by_finding_id`/`dismiss_inline_reply` require `GitHubProvider` specifically (the module's own docstring states GitHub-only — GitLab/Bitbucket have no F-ID/id-map system), the subcommand must call `_build_github_from_env()` (`vcs/__init__.py:99-122`) directly, or fail closed with a clear error if `VCS_PROVIDER` is set to anything other than `github`/unset. Do not silently no-op on non-GitHub providers — this command is GitHub-only by construction and should say so.
4. When `--finding-id` is provided, the subcommand calls `dismiss_by_finding_id(provider, finding_id, actor=actor, command=command)` and prints `result.reply` to stdout (matching the `slash` subcommand's `echo`-to-stdout convention at `cli.py:497-498`, which the workflow already captures via `reply=$(ai-pr-review slash ...)` — see `slash-commands.yml:1629-1672`'s exact pattern to mirror for the new job step).
5. On a `BODY`-classified finding (i.e. `result.feedback_source`/`result.feedback_file` are non-empty — `DismissResult` has no explicit "was this a body finding" flag, so gate on `feedback_source != "" or feedback_file != ""` per the dataclass fields at `dismiss.py:51-62`), the subcommand builds a `FeedbackEntry` (`feedback/models.py:17-44`: `ts`, `command`, `reason`, `source`, `file`, `rule_id`, `extras`) and writes it via `make_store(config).append(entry)` (`feedback/store.py:480-509`, `store.py:111`), gated on `AI_FEEDBACK_LOOP` / feedback-loop-enabled the same way the `slash` command's caller already is (check `cli.py`'s existing feedback-loop gating logic in the `slash` command body, `cli.py:437-499`, and reuse the same env var / condition — do not invent a new gate name). This folds the currently-separate "Record false-positive in feedback store" container-action step (`slash-commands.yml:880-910`) into the CLI call, closing the dead-`SLASH_*`-env gap noted in the approved plan (that step's env vars are set but never consumed by anything in Python today).
6. When `--finding-id` is omitted (list-active-IDs mode), the subcommand calls `list_active_body_ids(bodies)` (bodies sourced the same way `dismiss_by_finding_id` sources them internally — via `provider.list_bot_reviews()`) and prints a syntax-hint message to stdout listing the active `F<n>` IDs, matching the tone/content of the existing bash step's hint comment (`slash-commands.yml:466-515`).
7. Exit codes follow the `slash` command's documented convention (`cli.py:433-436`): `0` for handled (including "not found"/"nothing to dismiss" outcomes — these are not command failures), non-zero only for a genuine CLI usage error (missing required option) or an unrecoverable provider-construction failure (e.g. `VCS_PROVIDER` set to a non-GitHub value, or missing `GH_TOKEN`).
8. `.github/workflows/slash-commands.yml`'s `dismiss-body-finding` job (currently `runs-on: ubuntu-latest` with no `container:` key, spanning lines 412-1066) is converted to run inside the container, matching `feedback-command`'s exact pattern (`slash-commands.yml:1378-1383`):
   ```yaml
   runs-on: ubuntu-latest
   container:
     image: ghcr.io/tag1consulting/ai-pr-review:${{ inputs.image-tag }}
   defaults:
     run:
       shell: bash
   ```
9. Inside the converted job, steps 2 (`lookup`), 3 (`resolve inline finding + dismiss`), 4 (`record false-positive`), and 5 (`dismiss if all findings resolved`) — currently four separate steps totaling ~600 lines of bash/GraphQL/Python-heredoc (`slash-commands.yml:431-1013`) — collapse into a single step that runs `reply=$(ai-pr-review dismiss ...)`, following the exact invocation pattern already established in `feedback-command`'s "Invoke Python slash handler" step (`slash-commands.yml:1629-1672`): env vars set on the step, output captured to a shell variable, stderr redirected to a temp file for error surfacing, exit code checked.
10. Steps 1 (React — seen), 6 (React — done + reply comment), and 7 (React — not applicable) remain in YAML as workflow-level UX (reaction posting, comment posting) — these are not orchestration logic and are explicitly out of scope for the Python port per the approved plan.
11. The job's trust-gate `if:` condition (`slash-commands.yml:413-417`) is preserved unchanged.
12. `tests/python/test_cli.py` (or a new `tests/python/test_cli_dismiss.py` if the existing file's conventions don't fit — check `test_cli.py`'s current structure before deciding) gains test coverage for the new subcommand: `--finding-id` provided + BODY classification → feedback entry written + reply printed; `--finding-id` provided + INLINE classification → reply printed, no feedback entry; `--finding-id` omitted → active-IDs list printed; `VCS_PROVIDER` set to `gitlab`/`bitbucket` → clean failure with a GitHub-only error message, not a silent no-op; missing required option → non-zero exit.
13. `actionlint .github/workflows/slash-commands.yml` passes clean (this is a workflow-touching change — CLAUDE.md's release workflow calls for this check after touching workflow YAML).
14. `pytest tests/python -q`, `mypy ai_pr_review/`, and `ruff check ai_pr_review/ tests/python/` all pass clean. No regression in `tests/python/slash/test_dismiss.py` or `tests/python/vcs/test_dismiss_github.py` (13-1's test files) — this story adds a CLI wrapper around those functions, it does not change their behavior.
15. **Live-e2e smoke test before merge** (per the approved plan's delivery strategy: "one live-e2e smoke per phase, not a substitute for it"): on the seeded `tag1consulting/ai-pr-review-test` repo, exercise the out-of-diff, in-diff-body, and inline dismiss cases via the actual `/ai-pr-review dismiss F<n>` comment flow and confirm the new container-based job produces the same user-facing outcome (reply text, thread resolution, review dismissal) as the bash it replaces did.

### Debug Log References

- Provider construction: used `provider_from_env()` + explicit `VCS_PROVIDER` string gate (checked before calling `provider_from_env()`, not just an `isinstance` check after) rather than promoting `_build_github_from_env()`. `_build_github_from_env()` never inspects `VCS_PROVIDER`, so calling it directly would silently build a GitHub provider even when `VCS_PROVIDER=gitlab` — the exact silent-no-op AC3 forbids. An `isinstance`-only gate (checked after calling `provider_from_env()`) was also insufficient: `provider_from_env()` dispatches eagerly to `_build_gitlab_from_env()`/`_build_bitbucket_from_env()`, which raise their own provider-specific "token required" errors before an `isinstance` check would ever run, producing a confusing message on the non-GitHub path instead of AC3's mandated "GitHub-only" error. Fix: read `VCS_PROVIDER` (normalized exactly as `provider_from_env()` does: `(os.environ.get("VCS_PROVIDER") or "github").strip().lower()`) and fail closed with the GitHub-only message *before* calling `provider_from_env()`; the subsequent `isinstance` check is now unreachable dead-code-as-safety-net, kept for mypy narrowing.
- `make_store`/`FeedbackEntry` write path: no separate "feedback-loop gate" exists in `cli.py` to reuse as AC5/Task 2 assumed — verified `slash`'s body (`cli.py:394-499`) always calls `make_store(config)` and always attempts `store.append(entry)` unconditionally; `make_store` itself only gates on `vcs_provider`/`GH_TOKEN`/`GITHUB_REPOSITORY` presence, never on `AI_FEEDBACK_LOOP`. Deviation from the story text: added a new `--enable-feedback-loop` (`envvar=AI_FEEDBACK_LOOP`) CLI option and gated the `append()` call on it explicitly, because the bash job's step 4 ("Record false-positive in feedback store") *was* gated on `inputs.enable-feedback-loop == 'true'` — collapsing steps 2-5 into one CLI call would otherwise silently drop that gate and start writing to the long-lived feedback branch for repos that have the loop disabled. This is a repo-wide side effect, not a per-call one, so the gate was preserved rather than assumed away.
- Reply-honesty ordering (recurrence of the #556/PR-review F1-F3 bug class): `dismiss_by_finding_id`'s `DismissResult.reply` for a BODY finding is generated unconditionally as "...will be suppressed on future review runs," with zero knowledge of whether the caller's subsequent `store.append()` will succeed. The CLI must call `append()` first and choose its own wording from the boolean return (matching `handlers.py:107-125`'s existing pattern) rather than echoing `result.reply` before knowing the store write succeeded. Covered by `test_body_finding_feedback_store_failure_is_reported_honestly`.
- `explain`/`revise` commands (out of the CLI's `click.Choice`, but in the job's own trigger set) still route through the old bash `Lookup body finding or list active IDs` step, narrowed via `if: contains(fromJSON('["explain","revise"]'), ...)` — porting those two stub commands to Python is out of scope for this story (they are unrelated "not yet implemented" placeholders that happen to share this job's trigger, not dismiss orchestration). The new CLI-invocation step is narrowed to the complementary set (`["dismiss","false-positive","wont-fix"]`), so the two step families are mutually exclusive and never double-fire reactions/replies for the same comment.
- Reaction-selection divergence from prior bash behavior (deliberate, not a regression): the old `lookup` step set `inline_fid` *before* attempting inline-thread resolution, which suppressed the 😕 reaction (`React — not applicable` was gated `inline_fid == ''`) even when the inline thread could not actually be found — that case posted "could not be located, no action taken" with a 👍-adjacent non-reaction. The new `::notice::reaction=` marker in the CLI reacts 😕 whenever nothing was actually resolved/dismissed/recorded (`thread_resolved`/`review_dismissed`/`feedback_source`/`feedback_file` all falsy), which now includes the inline-thread-not-found case. Judged an improvement (a no-op should read as a miss to the user), but it is a genuine behavior change from the bash job and is called out here rather than silently shipped as "parity."
- **AC15 live-e2e smoke: partially completed pre-merge, remainder deferred to `:dev` pre-tag validation.** Built the branch image locally (`ghcr.io/tag1consulting/ai-pr-review:49da85c`) and confirmed `ai-pr-review dismiss` is registered and behaves correctly via three real `docker run` invocations against the live `tag1consulting/ai-pr-review-test` PR #1 (real GitHub API, no mocks): (1) BODY dismiss (F2) with `AI_FEEDBACK_LOOP=false` — correct reply with disabled-note, exit 0, `::notice::reaction=done` on stderr, stdout contained only the reply text (verified stdout/stderr are never mixed — the base64-round-trip contract the workflow wrapper depends on); (2) BODY dismiss (F3) with `AI_FEEDBACK_LOOP=true` — correct reply, and independently confirmed via the GitHub API that the `ai-pr-review-bot` branch (which did not exist beforehand) was created and `.ai-pr-review/learnings.jsonl` contains the correct `FeedbackEntry` (`command="false-positive"`, canonicalized from the raw `dismiss` alias; `source="semgrep"`, `file="lib/utils.go"`, `extras.finding_id=3`) — this is the specific regression the `--enable-feedback-loop` gate exists to prevent, and it round-trips correctly; (3) unknown F-id (F9999) — correct "could not find" reply, exit 0, `::notice::reaction=confused`; (4) list-active-IDs mode — correctly lists F1-F4 from the live review body (F1 still listed despite being dismissed in an unrelated prior session, which is expected: BODY dismissal is tracked via the feedback store, not by editing the static review body — a documented limitation from story 13-1).
  - **Not covered by the above, and not exercised before this merge:** the real `/ai-pr-review dismiss F<n>` PR-comment flow through the converted `container:` job (the YAML wrapper's env forwarding, base64 reply round-trip, and reaction-emoji step), and the `explain`/`revise` command family (which never invokes the new CLI — it stays on the preserved, narrowed `lookup` bash step).
